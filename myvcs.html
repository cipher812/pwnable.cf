Creating a versioning system

Version control systems are a category of software tools that help a software team manage changes to source code over time. Version control software keeps track of every modification to the code in a special kind of database. If a mistake is made, developers can turn back the clock and compare earlier versions of the code to help fix the mistake while minimizing disruption to all team members.Version Control System allows us to keep track of code, view,maintain and switch between different versions of the code. Benefits of version control systems. Developing software without using version control is risky, like not having backups. Version control can also enable developers to move faster and it allows software teams to preserve efficiency and agility as the team scales to include more developers.

Actions to be implemented:

  • Important: create, commit, delete, log, checkout
  • Optional: status, edit, rename, move, diff, revert, branch, merge, resolve, lock

Types:

  • Centralised (we'll go with this for the time being)
  • Distributed: ArX, Bitkeeper, DCVS, Git, Mercurial, Veracity, Subversion (refer if needed)

IMPORTANT : Read the Git Parable

How to:

First, we need a repository and a working directory(here we will keep our current code) Repo : Named .repo, this will be a hidden directory, which will contain all the files.

First, initialise the directory, using some command init.sh <repo_name>

Requirements:

  • Single argument(repo_name); exit with error if false
  • Check if the target is an existing file, if not, create a directory <repo_name>; exit with error if false
    • If target folder contains .repo, exit with error
    • If the above conditions are met, create the repo, with
      • .repo folder inside it
      • tags folder inside .repo
      • snapshots folder inside .repo

Now, for commit, check that we are inside the repo

  • Error if false

  • If true, copy the working files(except .repo), to the .repo/snapshots/ where number will be the ith no. of files

    .repo/snapshots/1 will be the first commit, .repo/snapshots/2 will be the second commit and so on.

  • Create a file called .commit inside .repo, which should include username, time and message of the person who made the commit

For log, create a log.sh file, which will store all the data about each commit(commit number,username,time,message)

If possible, include a .ignore file, which will ignore files that are not to be commited.
